<objective>
Close the visual loop for ASM-01: (a) thread the per-instance assemble pose into the
renderer so an assembled volume actually renders at its moved/rotated/scaled pose on the
assembly canvas, and (b) add the transform-mode selector + gizmo drag-signal wiring to
AssemblePage.qml so the user can pick Move/Rotate/Scale and drag the gizmo. After this
plan the feature is visually end-to-end (gizmo drag -> assemble transform written -> volume
re-renders at new pose). Plan 04 then locks in the 3MF round-trip.

Depends on Plan 01 (assemble accessors) and Plan 02 (routing) — the renderer reads the
assemble pose the routing now writes, and the QML forwards the same drag signals.
</objective>

<truths>
- T-01: The mesh blob the renderer consumes (`PrepareSceneData::modelVertices()`, produced by `EditorViewModel::meshData()` via `ProjectServiceMock::meshData()`) already bakes the ordinary per-object `m_transformation` (offset/rotation/scale) into world-space vertices. The assemble pose is NOT in this blob — it is a separate transform upstream stores on `m_assemble_transformation`. So the assemble pose must be applied as an additional per-object transform in the renderer's AssembleView path, parallel to how the explosion offset is already applied.
- T-02: `RhiViewportRenderer::buildModelVertices` (cpp:1608-1680) already has a `CanvasAssembleView` post-process branch (cpp:1617-1677) that applies the per-volume explosion offset to the baked vertices. This is the natural, lowest-risk place to also apply the assemble pose: after the explosion offset, apply the per-object assemble offset (translation) to every vertex of each object's batches. Rotation/scale of the assemble pose is a stretch goal — the Move gizmo is the primary ASM-01 interaction; rotate/scale assemble rendering can compose the full matrix if straightforward, else translate-only rendering with full transform persistence (Plan 01/02 already persist all three) is acceptable for this phase and noted as a render-fidelity follow-up.
- T-03: The renderer is a pure presentation layer (project rule: no business logic in QML/render). It must READ the assemble pose from the ViewModel/service, never compute or store it. The ViewModel exposes the pose; the renderer applies it. Mirror how `explosionRatio` already flows: QML binds `editorVm.explosionRatio` -> `RhiViewport` setter -> renderer member -> `buildModelVertices`.
- T-04: AssemblePage.qml currently binds `gizmoMode` ONLY to the Assembly Measure toggle (AssemblePage.qml:123-124). It has no transform-mode selector and no `onGizmoMoveRequested/onGizmoRotateRequested/onGizmoScaleRequested/onGizmoDragBegin/onGizmoDragEnd` handlers. PreparePage.qml:1747-1779 is the exact wiring template to mirror.
- T-05: PreparePage.qml has keyboard shortcuts (W/E/R) and a `setGizmoIfAvailable` helper that checks `availableGizmoMask` before switching mode (PreparePage.qml:127-134, 322-333). AssemblePage should mirror at least the mode-switch buttons gated on `availableGizmoMask` so the Plan 02 mask widening actually lights up the UI.
- T-06: The renderer needs the per-object assemble offset vector keyed by `sourceObjectIndex`. The ViewModel must expose this as a Q_PROPERTY or Q_INVOKABLE that the QML binds into the viewport (e.g. `assembleOffsets` list), paralleling `meshBatchSourceObjectIndices`. Do not have the renderer call the service directly.
- T-07: When a gizmo drag writes the assemble transform (Plan 02), `emit stateChanged()` fires; the QML bindings re-evaluate; the viewport's assemble-offset property updates; `update()` triggers `synchronize()+render()`; `buildModelVertices` re-applies the new offset. This is the same re-render loop the explosion ratio uses (AssemblePage.qml:109-114 comment).
</truths>
