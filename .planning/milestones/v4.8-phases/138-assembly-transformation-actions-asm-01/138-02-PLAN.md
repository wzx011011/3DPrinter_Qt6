<objective>
Wire the assembly-canvas transform routing in EditorViewModel so the existing
canvas-agnostic Move/Rotate/Scale gizmo apply-slots target the assemble transform
when `m_activeCanvasType == 2` (CanvasAssembleView), and advertise the move/rotate/scale
gizmo bits on AssembleView. This is the routing half of ASM-01 — no new gizmo
machinery, just branching the existing slots.

This plan is independent of Plan 01 at the file level (different files) BUT its
task 2+ acceptance (the slots calling `setAssembleOffset` etc.) requires Plan 01's
accessors to exist. Run after Plan 01 if executing serially, or merge the accessor
calls in once Plan 01 lands. The `availableGizmoMask` flip (task 1) has no such
dependency.
</objective>

<truths>
- T-01: The existing Move/Rotate/Scale gizmo rendering (RhiViewportRenderer.cpp:1862-1920) and drag signals (RhiViewport.h:314-316 `gizmoMoveRequested/Rotate/Scale`) are canvas-agnostic — they fire on whichever canvas is current. No new gizmo rendering or signals are needed.
- T-02: `availableGizmoMask()` hard-returns `0` (or bit 19 for Assembly Measure only) when `m_activeCanvasType == 2` (EditorViewModel.cpp:3178-3183). This is THE gate preventing the move/rotate/scale gizmos from activating on AssembleView. It must be widened to advertise bits 0/1/2 (GizmoMove/Rotate/Scale) when a single volume/object is selected on AssembleView, while preserving the existing Assembly Measure bit-19 behavior.
- T-03: `canActivateGizmo(0/1/2)` currently returns `hasSingleObject` for Move/Rotate/Scale (EditorViewModel.cpp:5749-5756). On AssembleView the same single-selection rule applies, so once `availableGizmoMask` advertises the bits, `canActivateGizmo` already permits them — no change to `canActivateGizmo` should be needed. Verify this; only add an AssembleView-aware branch if the existing logic proves insufficient.
- T-04: The apply-slots `applyGizmoMoveDelta`/`applyGizmoRotateDelta`/`applyGizmoScaleFactor` and their `begin*Drag`/`end*Drag` counterparts (EditorViewModel.cpp:347-615) UNCONDITIONALLY call `setObjectPosition/Rotation/Scale` and push a `TransformCommand`. On the assembly canvas these must instead call `setAssembleOffset/Rotation/Scale` and push an `AssembleTransformCommand` (from Plan 01). The Prepare path (`m_activeCanvasType != 2`) stays byte-for-byte unchanged.
- T-05: Upstream behavior: on the assembly canvas the Move/Rotate/Scale gizmo writes `ModelInstance::m_assemble_transformation` (Model.hpp:1280-1294); the ordinary `m_transformation` is untouched. The Qt routing must match this — NEVER write both.
- T-06: The drag-begin slots capture the starting transform for the undo command (`m_gizmoMoveDragStartPos`, `m_gizmoRotateDragStartRot`, `m_gizmoScaleDragStartScale`). On the assembly canvas these must capture the ASSEMBLE transform, not the ordinary transform, so the undo delta is computed against the correct baseline.
- T-07: The `invalidateSliceResultsForCurrentPlate()` call in the apply slots is a Prepare-path concern (it dirties slice cache for moved objects). On AssembleView the assemble transform does not invalidate slice results (it is a non-sliced pose), so that call should be skipped when `m_activeCanvasType == 2`. Preserve it on the Prepare path.
</truths>
