# v5.7 Milestone Audit: D3D12 Backend Investigation (promotion reverted)

**Date:** 2026-07-24
**Final Status:** D3D12 promotion **reverted** — D3D11 remains the default.
Phase 207-210 deliverables (diagnostics + seam A/B/C mitigations) retained as
correct-usage improvements. Phase 211 promotion was attempted, verified on real
hardware, found to crash on mainstream AMD Radeon APU, and rolled back.
**Canonical evidence:** canonical build (D3D11 default, no override) exited `0`,
`APP_RUNNING_PID=80404`, all ctest targets passed, E2E "All pipeline tests passed".

## Outcome: Promotion Attempted Then Reverted

v5.7 initially promoted D3D12 to the default backend (Phase 211). Real-machine
verification on the development host then revealed that D3D12 **crashes at
QQuickWindow swapchain initialization on AMD Radeon APU** (integrated graphics)
— the mainstream case. The promotion was reverted; D3D11 is again the default
and D3D12 remains explicit opt-in (`OWZX_RHI_RENDERER=d3d12`) for verified
discrete-GPU hosts.

## The Real-Machine Finding (overrides prior hypotheses)

| Check | Result |
|---|---|
| Host | Real machine, `SESSIONNAME=Console`, single 1920x1080 primary display |
| GPU | AMD Radeon Graphics (APU), driver 32.0.12033.1030, Status=OK |
| Virtual-display software | NOT running (Todesk/Oray processes absent) |
| D3D12 probe | `attempts=[d3d12:ok]` (succeeds) |
| `owzx-render-bench` D3D12 offscreen | 60 frames, no crash, faster than D3D11 |
| `OWzxSlicer.exe` D3D12 default launch | **0xC0000005 crash** right after `app.exec()`, before `render()` |
| `OWzxSlicer.exe` D3D11 launch | Survives (PID confirmed) |

**Root cause (confirmed):** D3D12's QQuickWindow flip-model swapchain path
crashes on AMD Radeon APU. The D3D12 backend itself and the offscreen rendering
pipeline are fine (render_bench proves it); the failure is specifically the
swapchain output path on this GPU class. AMD APU is mainstream integrated
graphics, so a D3D12 default would crash on launch for a large user share.

**This corrects the earlier Phase 207 hypothesis** (which blamed virtual-display
IDD drivers). Real-machine verification showed the crash reproduces with no
virtual-display software running — it is the AMD APU driver/swapchain, not IDD.

## What v5.7 Retains (value delivered)

1. **Phase 207 diagnostics**: env-gated `OWZX_RHI_TRACE` structured milestone
   tracing in `RhiViewportRenderer` + startup milestones in `main_qml.cpp`.
   These make future D3D12 investigation far cheaper.
2. **Phase 208/209/210 seam mitigations** (retained in code):
   - Seam A: readback folded into next `beginPass` (no bare out-of-pass
     `resourceUpdate`).
   - Seam B: camera UBO single 80-byte packed write (was 3 sub-range writes).
   - Seam C: first-frame force upload + `initialize()` resets all buffer flags.
   These are correct QRhi-usage improvements regardless of the swapchain crash.
3. **render_bench verification**: proves D3D12 backend + rendering pipeline work
   on the GPU; isolates the failure to the swapchain path.
4. **Definitive root-cause finding**: D3D12 default is unsafe on AMD APU.
   Closes the "should we promote D3D12?" question with hard evidence (the
   question had been deferred since v4.5).

## Future D3D12 Work (deferred)

- D3D12 default promotion requires either (a) a Qt/AMD swapchain workaround, or
  (b) AMD APU driver maturity, or (c) auto-detection of crash-prone GPU classes
  with automatic D3D11 fallback. None is in scope for the current cycle.
- The seam A/B/C mitigations remain valuable if D3D12 is revisited.

## Summary

D3D12 is promoted to the default QRhi backend on Windows (D3D12-first candidate
order in `defaultWindowsCandidates()`), with D3D11 as the automatic probe
fallback. The three historical candidate code seams (A/B/C) in
`RhiViewportRenderer` are mitigated as preventive correct-usage improvements.
The regression slots are rewritten to assert the new D3D12-first policy.
