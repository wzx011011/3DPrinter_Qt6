<domain>
## Phase Boundary

Port the rotate and scale gizmos onto the default D3D11 RHI path. This phase
delivers visible rotate torus rings, visible scale shafts with box handles,
axis picking through the Phase 65 `GizmoMath` helpers, and drag interaction
that updates the selected object's rotation or axis scale through
`EditorViewModel`.

In scope:
- RHI upload/draw of rotate and scale geometry generated by Phase 66
  `GizmoGeometry`.
- `RhiViewport` rotate/scale pick and drag lifecycle using
  `GizmoMath::pickRotateAxis`, `GizmoMath::pickScaleAxis`,
  `GizmoMath::computeRotateAngle`, and `GizmoMath::rayToAxisT`.
- `EditorViewModel` begin/apply/end methods for rotate and scale drags,
  matching Phase 69's one undo entry per drag.
- Thin QML signal forwarding in `PreparePage.qml`.
- Focused viewmodel and QML audit tests plus canonical verification.

Out of scope:
- Full OrcaSlicer `GLGizmoScale3D` grabber-plane ratio and Ctrl-constrained
  pivot behavior. The current v3.8 path follows the extracted GLViewport
  helper semantics already in this repository; richer upstream scale parity
  should be tracked as future source-truth work.
- Hover/highlight color rendering.
- Cut plane and wipe tower rendering, which remain Phase 71.
- Precise object picking, which remains Phase 72.

<decisions>
## Implementation Decisions

### Rendering Shape
- Reuse `GizmoGeometry::buildRotateGizmoVertices()` and
  `GizmoGeometry::buildScaleGizmoVertices()` rather than adding new geometry
  generation in the renderer.
- Keep one shared gizmo shader and the existing RHI gizmo line/triangle
  pipelines. Rotate rings and scale boxes draw with the triangle pipeline;
  scale shafts draw with the line pipeline.
- Store `GizmoGeometryOffsets` in `RhiViewportRenderer` after upload so draw
  calls use builder-owned offsets instead of hardcoded rotate/scale counts.
- Preserve Phase 68 no-depth-write gizmo behavior so rotate/scale handles stay
  visible through objects like the GL path's depth clear.
- Defer hover/highlight colors; base axis colors are sufficient for Phase 70.

### Interaction Semantics
- `RhiViewport` owns all pick and ray math. QML forwards signals only.
- Rotate drag uses `GizmoMath::computeRotateAngle()`. The math helper returns
  radians; `EditorViewModel` converts deltas to degrees before applying them
  to the existing object rotation state.
- Rotate and scale use per-frame delta signals with begin/end bracketing, then
  `EditorViewModel` records one `TransformCommand` for the whole drag.
- Scale drag follows the repository's extracted GLViewport path:
  `pickScaleAxis`, `rayToAxisT`, scale factor `1.0 + delta * 0.01`, clamped to
  `0.01`, applied only to the picked axis.
- Scale geometry still renders visible box handles; pick precision is tied to
  the current `GizmoMath::pickScaleAxis` helper for this phase.

### Verification Scope
- Add focused `ViewModelSmokeTests` coverage for rotate and scale drag
  coalescing.
- Extend `QmlUiAuditTests` so the C++ ownership boundary covers rotate and
  scale signals, math, renderer geometry upload/draw, and thin QML forwarding.
- Run focused tests first, then run the canonical verifier before closing the
  phase because this phase changes C++ and QML behavior.

### the agent's Discretion
- Exact private helper names in `RhiViewport` and `RhiViewportRenderer`.
- Whether rotate/scale signals are separate or share begin/end with mode-
  specific apply signals, as long as QML stays thin and tests pin behavior.
- Exact plan/task split, provided each task remains testable.

</decisions>

<specifics>
## Specific Ideas

- Render all three gizmo shapes from one static RHI vertex buffer by appending
  move, rotate, and scale vertices and adding per-shape base offsets to the
  builder offsets.
- Keep rotate deltas axis-indexed as 1=X, 2=Y, 3=Z to match the existing
  `GizmoMath` API and Phase 69 move axis convention.
- Keep scale deltas axis-indexed and pass a clamped scale factor to
  `EditorViewModel` rather than raw ray-axis distance.
- Use one-command undo macros for rotate and scale drag end, matching Phase
  69's protection against merging into the previous transform command.

</specifics>
