# Phase 91 Plan: Explosion Ratio And Assembly Rendering

## Objective

Add the explosion-ratio control + per-volume separation rendering + yellow dashed
connector guide lines to the Phase 90 `CanvasAssembleView` canvas host on the
default RHI/D3D11 path, without regressing Prepare/Preview.

This is the second phase of v4.2 with production code. The deliverable proves the
explosion works end-to-end: load a multi-part model, switch to AssembleView, drag
the "爆炸比例" slider, and see the volumes separate radially along their assembly
axes with yellow dashed connector guide lines between them (matching
`shotScreen/装配页_爆炸.png`), plus a reset-to-default action. The Assembly
measurement gizmo (Phase 92), the data pool (Phase 93), and runtime pixel proof
(Phase 93) are out of scope here.

## Requirements

- `ASMEXPLODE-01`: User can adjust an explosion-ratio control that separates
  volumes of multi-part objects for assembly inspection, mirroring upstream
  `m_explosion_ratio` behavior (default 1.0, reset capability). Covers matrix
  region ASM-EXPLODE-SLIDER.
- `ASMEXPLODE-02`: AssembleView renders all loaded model volumes on the default
  RHI/D3D11 path with correct per-volume separation at the current explosion
  ratio, without regressing Prepare/Preview rendering. Connector guide lines
  between separated parts render when the ratio is non-default. Covers region
  ASM-EXPLODE-RENDER.

## Per-Volume Granularity (key implementation finding)

The current mesh blob in `ProjectServiceMock::meshData()` (line 5868) emits one
batch per instance, flattening all volumes of an object into a single batch. This
makes per-volume explosion offset impossible.

Phase 91 restructures the blob to emit one batch per **volume** (each batch still
tagged with its parent object's `sourceObjectIndex`). This is safe because
`RhiViewportRenderer::uploadHighlightBuffer()` (line 584-652) already unions the
bounds of ALL batches matching a given `sourceObjectIndex`, so Prepare's
selection highlight is byte-for-byte identical with per-volume batches. The
per-volume batches expose each volume's center via `ModelBatch::bounds`, which the
renderer uses to compute the explosion offset.

This change is the source-truth-correct approach: upstream
`GLVolumeCollection` stores one `GLVolume` per volume, and `m_explosion_ratio`
offsets each volume independently (the per-volume offset is the whole point of
the feature per `装配页_爆炸.png`).
