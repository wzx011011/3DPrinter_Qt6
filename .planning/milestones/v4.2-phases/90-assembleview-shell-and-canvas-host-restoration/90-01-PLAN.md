# Phase 90 Plan: AssembleView Shell And Canvas Host Restoration

## Objective

Replace the `Plater.qml` AssembleView placeholder (`assembleSlot` Item + "иЈ…й…Ќи§†е›ѕжљ‚дёЌеЏЇз”Ё"
Text) with a real AssembleView page shell + canvas host registered as the third
`CanvasAssembleView`-equivalent canvas on the default RHI/D3D11 path, plus the
navigation entry, screenshot-aligned 4-region chrome, and Plater-level
`CanvasAssembleView` routing branches вЂ” without breaking Prepare/Preview.

This is the first phase of v4.2 with real C++/QML production code. The deliverable
proves the canvas host works end-to-end: load a model, switch to AssembleView, and
see the mesh rendered. Explosion-ratio separation (Phase 91), the Assembly gizmo
(Phase 92), the data pool (Phase 93), and runtime pixel proof (Phase 93) are out of
scope here.

## Requirements

- `ASMSHELL-01`: User can navigate to a real AssembleView page (replacing the
  `Plater.qml` placeholder) from the intended entry point(s), and the page renders
  as a screenshot-aligned canvas host with no leftover "temporarily unavailable"
  text. Covers matrix regions ASM-SHELL, ASM-ASSEMBLY-INFO, ASM-NAVIGATION.
- `ASMSHELL-02`: AssembleView is registered as the third `GLCanvas3D`-equivalent
  canvas host (mirroring `CanvasAssembleView = 2`) and coexists with Prepare/Preview
  without breaking navigation, payload, or view-mode state. Covers region ASM-CANVAS.
- `ASMROUTE-01`: AssembleView selection, undo/redo, and gizmo routing branch
  correctly on `CanvasAssembleView` (mirroring upstream `Plater.cpp` conditionals)
  so Prepare/Preview behavior is unchanged while AssembleView gets its own routing.
  Covers region ASM-PLATER-ROUTING.

## Locked Decisions (from 90-CONTEXT.md вЂ” MUST honor)

1. **Canvas:** add `CanvasAssembleView = 2` to the existing `RhiViewport::CanvasType`
   enum (`RhiViewport.h:60-65`). NO separate `AssembleViewport` class.
2. **Model data:** share via existing `EditorViewModel` / `ProjectServiceMock`. NO
   separate scene copy.
3. **Renderer:** extend `RhiViewportRenderer` with a `CanvasAssembleView` branch
   (basic mesh render). NO dedicated `AssembleViewportRenderer`.
4. **Shell:** new `AssemblePage.qml` with 4-region chrome. NOT embedded in
   `Plater.qml`.
5. **Navigation:** view-mode toggle in the top toolbar (`BBLTopbar.qml`) bound to
   `backend.requestChangeViewMode(backend.vmAssembleView)`.
6. **Sidebar:** reuse `LeftSidebar.qml` with AssembleView visibility filtering.
7. **Routing:** `canvasType == CanvasAssembleView` branches in
   `BackendContext`/`EditorViewModel` mirroring `Plater.cpp:7322,11601,11635,11744,11823`.
   Prepare/Preview paths stay untouched.
8. **Undo/redo:** shared `UndoRedoManager` (single stack).
9. **Explosion slider:** DEFERRED to Phase 91. Bottom controls show assembly-info
   panel only.
10. **Basic mesh rendering:** YES in Phase 90 (proves the canvas host works
    end-to-end).
