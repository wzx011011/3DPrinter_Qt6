# Roadmap: v5.7 D3D12 Default-Backend Promotion

## Goal

Promote D3D12 from explicit opt-in to the default QRhi backend on Windows,
behind an automatic D3D11 fallback safety net. Before promotion, reproduce and
locate the historical `0xC0000005` crash as far as the environment allows, and
apply the three candidate code mitigations (seams A/B/C).

**Core Value:** clear the longest-standing deferred item (D3D12 default-backend
promotion) by first attempting root-cause confirmation in this environment,
then mitigating the three hypothesized seams, then promoting with a fallback
so a residual D3D12 crash degrades to D3D11 rather than crashing the app.

## Scope Rule

- **Allowed:** RhiBackendSelector candidate reorder + automatic fallback,
  RhiViewportRenderer diagnostic logging, seam A/B/C mitigations, regression
  slot rewrites, planning doc updates.
- **Forbidden:** modifying libslic3r; patching upstream OrcaSlicer source;
  alternate build directories/scripts; embedding Python; reopening
  LAN/device/cloud scope.
- **Canonical build command:** `powershell -ExecutionPolicy Bypass -File scripts/auto_verify_with_vcvars.ps1`
- **Canonical build directory:** `build/`

## Background (why promotion was deferred)

v4.5 (Phase 106) and v5.6 (Phase 203) both documented a `0xC0000005`
STATUS_ACCESS_VIOLATION crash ~3s after launch under D3D12 on the original
developer workstation (2026-06-27). The crash never reproduced in the headless
test environment (two attempts, 12s/20s, both survived). Root cause was never
confirmed. Three candidate code seams (A/B/C) were hypothesized but never
mitigated. The historical minidumps no longer exist, so offline WinDbg
analysis is impossible. v5.7 closes this by reproducing as far as possible in
this environment, mitigating all three seams, then promoting with a fallback.

## Verification Gate

Each phase:
- [ ] Canonical build exits 0.
- [ ] Relevant ctest targets PASS (including rewritten D3D12 slots).
- [ ] `OWzxSlicer.exe` launch liveness (PID reported).
- [ ] Regression slots intact (rewritten D3D12-first assertions).

Phase 207 extra: in-environment D3D12 run >=30s startup_diagnostics.log
evidence (survived or crash stack).
Phase 211 extra: D3D12 launch survives + fallback verified (probe D3D12 fail
-> D3D11).

## Known Limitation (out of environment)

Final confirmation that the historical crash is gone requires the original
developer machine (real GPU + display) running the mitigated build under
`OWZX_RHI_RENDERER=d3d12`. If it still crashes there, a native debugger
(VS/WinDbg) attached to the crashing process captures the faulting stack.
This is recorded in Phase 211 PLAN.md as `pending_machine_verification`. The
automatic D3D11 fallback is the safety net if a residual D3D12 crash appears
in the field.
