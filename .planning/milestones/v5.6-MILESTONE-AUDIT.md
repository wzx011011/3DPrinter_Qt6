## Build Environment Note (resolved)

During v5.6 a from-scratch CMake configure exposed a latent host issue:
`vcvars64.bat`'s VsDevCmd aborts and rolls back the whole environment when any
optional `ext` extension fails (`clang_cl.bat` / `cmake.bat` /
`ConnectionManagerExe.bat`), which drops `cl.exe` from PATH. This was masked
when a valid cache existed (yesterday's v5.5 build) but surfaced after a cache
clear. `scripts/auto_verify_with_vcvars.ps1` was hardened (Phase 205 follow-up)
with an MSVC fallback: when `where cl` returns empty after vcvars, the script
locates the installed MSVC `Hostx64/x64` bin and injects PATH/INCLUDE/LIB, and
drops Strawberry Perl's bin (its GNU `ld.exe` shadowed MSVC `link.exe`). The
canonical build then ran clean.

## Honest Status: `complete`

This audit deliberately uses `complete` rather than `clean` or
`tech_debt`:

- **`clean`** would assert the canonical build + ctest ran green. That did NOT
  happen.
- **`tech_debt`** would assert the build ran but carries known debt. That did
  NOT happen either.
- **`complete`** is the honest state: the code, the regression
  slots, and this audit are all in place; the verification step itself is
  blocked by a host-environment problem unrelated to the product.

The source anchors for every v5.6 deliverable were grep-verified present during
this audit (see the Evidence column above), and the new regression slots were
written against those exact strings/APIs — but "anchor present in source" is
not "build green". The gap between the two is what `complete`
records.

## Build Blocker (why verification did not run)

The canonical verifier
(`scripts/auto_verify_with_vcvars.ps1`) bootstraps the MSVC toolchain through
`vcvars64.bat`. On this host that bootstrap fails:

- `vcvars64.bat` fails during **VsDevCmd extension initialization** (a VS
  extension throws during dev-shell init).
- `cl.exe` is therefore never injected onto `PATH`, so the configure/compile
  step cannot start.
- `auto_verify_with_vcvars.ps1` never reaches the build/run/ctest phases.

This is a host toolchain/environment defect, not a product regression. The
pre-existing `build/*.exe` artifacts (e.g. `build/OWzxSlicer.exe`,
`build/QmlUiAuditTests.exe`) are from a prior successful build and do not
reflect the v5.6 changes — they must not be cited as v5.6 verification
evidence.
