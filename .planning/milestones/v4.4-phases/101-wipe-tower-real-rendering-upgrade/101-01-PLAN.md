# Phase 101 Plan: Wipe-Tower Real Rendering Upgrade

## Objective

Close WTRENDER-01/02 by confirming the Option A dimensioned-box baseline is
the v4.4 wipe-tower rendering contract, documenting Option B (real mesh) as a
future upgrade, and closing the SoftwareViewport rendering gap so the software
fallback path does not lag the RHI path.

**Key finding from pre-planning research:** the RHI render path is ALREADY
correct end-to-end for Option A — `RhiViewportRenderer::uploadWipeTowerBuffer`
(RhiViewportRenderer.cpp:1064-1095) already calls
`buildWipeTowerVertices(m_wipeTowerX, m_wipeTowerZ, m_wipeTowerWidth,
m_wipeTowerDepth, m_wipeTowerHeight)` with real dims, rebuilds on
`m_wipeTowerDirty`, and `synchronize()` (lines 171-189) already pulls the real
dims from the RhiViewport Q_PROPERTYs (which Phase 100 bound to editorVm). So
WTRENDER-01 on the RHI path needs NO structural change — it needs a regression
test proving the rebuild fires with the real dims + a source-audit-grade
comment locking Option A as the baseline.

**The one real gap is SoftwareViewport:** SoftwareViewport.cpp has zero
wipe-tower paint code (grep returns nothing). The Q_PROPERTY setters exist
(lines 207-253, each calls `update()`) but no `paint()` consumes them.
SoftwareViewport is the registered fallback `GLViewport` QML type
(main_qml.cpp:305) used when QRhi is unavailable. Two acceptable resolutions:
(a) add a minimal QPainter wipe-tower box in SoftwareViewport's paint matching
the RHI path's real dims (so the software fallback path does not silently
drop the wipe-tower), OR (b) document the gap as accepted for v4.4 with a
justification (SoftwareViewport is not the active Prepare renderer; the
default path is RhiViewport; the software path is a driver/init fallback).
This plan chooses (a) — the minimal QPainter box — because the work is small
(20-30 lines) and it makes the software fallback consistent with the RHI
path. The planner judged (b) acceptable too; the executor may defer with
justification if (a) proves non-trivial.

Phase 101 is primarily a verification + consistency phase. It does NOT
implement Option B (real mesh) — that is LOCKED as a future upgrade per
Frozen Decision 2.

1. **WTRENDER-01 (RHI path real-dims rebuild, regression test):** Add a test
   that feeds real dims through EditorViewModel → RhiViewport Q_PROPERTYs and
   asserts the renderer's `m_wipeTowerDirty` rebuild fires with them. The
   existing `uploadWipeTowerBuffer` already does the right thing; the test
   locks the contract so a future refactor cannot silently break it. Add a
   code comment in `buildWipeTowerVertices` locking Option A as the v4.4
   baseline.

2. **WTRENDER-02 (Option A baseline documented; Option B deferred):** Add a
   code comment block in `GizmoGeometry.cpp::buildWipeTowerVertices` citing
   the upstream Option A source-truth (`3DScene.cpp:840-885
   load_wipe_tower_preview` + `make_cube(width, depth, height)` at :855) and
   documenting Option B (`3DScene.cpp:887-925 load_real_wipe_tower_preview` +
   `convex_hull_3d()` at :914 from `wipe_tower_mesh_data->real_wipe_tower_mesh`
   + `real_brim_mesh` at Print.hpp:745-746) as a future upgrade requiring ITS
   vertex format extension in GizmoGeometry + RhiViewportRenderer.

3. **SoftwareViewport rendering gap:** Add a minimal QPainter wipe-tower box
   in `SoftwareViewport::paint` consuming the 6 Q_PROPERTYs (show gate +
   width/depth/height/x/z) so the software fallback path renders the real
   sliced wipe-tower, not nothing. Mirror the RHI box's color
   `{0.35, 0.60, 0.85, 0.50}` (GizmoGeometry.cpp:463).
