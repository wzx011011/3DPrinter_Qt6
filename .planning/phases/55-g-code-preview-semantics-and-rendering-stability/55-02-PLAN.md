<objective>
Close GCODE-01 (real-data path stays real, no placeholder) and GCODE-02 (full upstream 20-role parsing + 17 view modes + render-side per-role visibility) at the data + renderer contract level. UI binding to the new toggles is Plan 03; the audit-test extensions for GCODE-01/03/04/05 are Plan 04.

Purpose: The 55-RESEARCH.md identifies three concrete gaps in the current implementation: (1) `styleFor()` collapses `;TYPE:` into 5 coarse categories instead of upstream's 20 fine-grained role values; (2) StoredSegment/PackedSegment/GcvPackedSegment carry no role field, blocking per-role visibility filtering; (3) viewModes() returns 13 entries vs upstream's 17 EViewType values, with index 9 mislabeled "Flow" vs upstream "Filament". This plan closes all three by extending the GCV1 wire format in lockstep across viewmodel + renderer and replacing the coarse parser with the upstream-aligned fine-grained mapping.

**CRITICAL ENUM-DIVERGENCE FIX:** The libslic3r `ExtrusionRole` (parser-side, in `ExtrusionEntity.hpp`) and the libvgcode `EGCodeExtrusionRole` (color/visibility-side, in `Types.hpp`) have DIFFERENT orderings past index 6. The Qt6 codebase uses the **libvgcode `EGCodeExtrusionRole` index as canonical** everywhere — the parser maps the upstream display string DIRECTLY to the libvgcode index, never via the libslic3r integer. See 55-RESEARCH.md "Pattern 2" for the divergence cross-reference table. Without this fix, 10 of 20 roles get the wrong FeatureType-mode color and the wrong visibility slot.

Render-side role filtering (no repack) preserves the Phase 41 interaction-stability guarantee that CONTEXT.md locks as a hard constraint. Plan 02 also flips `showTravelMoves_` to default `false` to honor CONTEXT.md's "travel and wipe hidden after first view" and upstream's Travels/Wipes = false defaults.

Output: Modified PreviewViewModel (.h/.cpp), RhiViewport (.h/.cpp), RhiViewportRenderer (.h/.cpp). PreviewParserTests RED slots from Plan 01 flip to GREEN; new divergent-role-color test added.
