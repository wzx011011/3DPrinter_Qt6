### Issue found and resolved during execution (not a plan deviation — test-internal refinement)

**Initial E2E drivers were non-deterministic; refined to the proven pattern.**
- **Found during:** canonical build verification (Task 3 E2E run).
- **Issue 1:** `sliceFailedClearsPreviewState` originally seeded the preview via `loadGCodeFromPrevious(tests/fixtures/orca_sample.gcode)`, but OrcaSlicer rejects that fixture on the reuse path ("Not a gcode file generated by Orca Slicer"), so the seed failed. The `resliceRebuildsGcodePreviewDataWithDifferentBytes` test called `editor.requestSlice()` a second time, which is gated by stale-result checks and hung (300s timeout) then crashed the test process (0xc0000409).
- **Fix:** `sliceFailedClearsPreviewState` now seeds via a real live slice and triggers `sliceFailed` with `loadGCodeFromPrevious` on a non-existent path (deterministic early-fail branch). `resliceRebuilds...` now uses the proven `slice.startSlice()` + `slice.clearResults()` + reimport pattern (matching the existing `test_previous_gcode_reuse` / `test_import_invalidates` tests) and injects the layer_height override through the same minimal-config path.
- **Files modified:** tests/E2EWorkflowTests.cpp only.
- **Verification:** second canonical build run — `[E2E] All pipeline tests passed`, no crash, no timeout.
- **Committed in:** `f39db29`.

**Total deviations:** 0 source/build deviations. 1 test-internal refinement (test-only, no scope creep).

## Issues Encountered
- OrcaSlicer's G-code reuse path (`export_gcode_from_previous_file`) only accepts G-code files carrying the Orca "generated by Orca Slicer" header; the committed `tests/fixtures/orca_sample.gcode` is an Orca-*style* fixture (good for the parser/view-model tests in Plans 01/02 and ViewModelSmokeTests here) but is NOT a valid Orca-reuse input. Resolved by seeding E2E preview state via a real live slice instead.
- The canonical verify script launches `OWzxSlicer.exe` as a smoke test and runs E2E as explicitly non-blocking (yellow warning, never a script exit). The authoritative blocking gates are the per-suite `[...] tests passed` lines; E2E is reported but does not fail the build on test-level issues.
