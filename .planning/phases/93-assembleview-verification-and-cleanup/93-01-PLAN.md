## Scope Decision (data pool вЂ” what is ported vs. simplified) вЂ” EXPLICIT

Upstream `AssembleViewDataPool` (`GLGizmosCommon.hpp:274-295`,
`GLGizmosCommon.cpp:433-491`) is a small cache that holds two resources keyed by
`AssembleViewDataID`: `ModelObjectsInfo` (the live `ModelObjectPtrs` from the
scene) and `ModelObjectsClipper` (a per-volume `MeshClipper` set driven by a
clipping-plane ratio). It is updated from `GLGizmosManager::update_data()`
(`GLGizmosManager.cpp:427-431`) with a bitmask and consumed by
`GLGizmoAssembly`/`GLGizmosManager` for the cut-plane render (`:566-611`).

Phase 92 already computes Assembly-measure bounds directly from
`m_cachedMeshBatchSourceObjectIndices` + mesh blobs
(`EditorViewModel::selectedVolumeBoundsForAssemblyMeasure()`); it does NOT use a
pool. The full upstream pool depends on a `MeshClipper`, which in turn needs a
per-volume `indexed_triangle_set` (the GPU-oriented mesh blob is not an ITS) вЂ”
the same hard dependency that blocked the full feature-picking engine in Phase
92. Phase 93 therefore delivers a **minimal-but-correct cache that mirrors the
upstream pool's shape and isolation contract** and is consumed by the Assembly
gizmo path, with the `MeshClipper` resource documented as a deferred future
resource:

| Capability | Upstream | Phase 93 | Rationale |
|---|---|---|---|
| `AssembleViewDataID` enum (bitmask) | `GLGizmosCommon.hpp:268-272` | **FULL** | Source-truth enum shape; cheap; the bitmask + `update(required)` contract is the pool's API. |
| `AssembleViewDataBase` polymorphic base | `GLGizmosCommon.hpp:299-332` | **FULL** (the `update()`/`release()`/`is_valid()` lifecycle) | The base is what gives the pool its resource-management shape; mirroring it keeps the port faithful and leaves the door open for the clipper resource. |
| `AssembleViewDataPool::update(bitmask)` + getters | `GLGizmosCommon.cpp:441-468` | **FULL** | The `update(required)` release-what-is-not-used semantics is the pool's core behavior. |
| `ModelObjectsInfo` resource (live per-object info) | `GLGizmosCommon.cpp:496-509` (pulls `model->objects`) | **MATCHING** (cached per-object info: source-object index в†’ bounds в†’ center в†’ longest-axis, sourced from the existing `m_cachedMeshBatchSourceObjectIndices` + mesh blobs on the AssembleView path) | The Assembly gizmo needs per-object info; Phase 92 computes it on-demand вЂ” Phase 93 formalizes it as a cached pool resource fed into `selectedVolumeBoundsForAssemblyMeasure()`. |
| `ModelObjectsClipper` resource (cut-plane) | `GLGizmosCommon.cpp:518-559` (per-volume `MeshClipper` set) | **DEFERRED** (enum slot reserved; not registered; future) | Needs per-volume ITS вЂ” same hard dependency as the Phase 92 feature-picking engine. Documented as a future resource that slots into the same pool without an API change. |
| Isolation from Prepare/Preview | pool only updated/read on `CanvasAssembleView` (`GLGizmosManager.cpp:427-431` runs in the gizmo update which is AssembleView-only) | **FULL** вЂ” the cache lives in `EditorViewModel` behind the `m_activeCanvasType == 2` gate; Prepare/Preview never populate or read it | ASMROUTE-02's core constraint. |

**This is the documented simplification.** It honors ASMROUTE-02 (a real
`AssembleViewDataID`/`AssembleViewDataPool` cache that the Assembly gizmo
consumes, isolated from Prepare/Preview) while keeping the work proportional to
the screenshot evidence (the three AssembleView screenshots do not show a cut
plane). Everything deferred is recorded as a future resource that fits the same
pool API.
