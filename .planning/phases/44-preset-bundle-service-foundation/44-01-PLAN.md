# Phase 44 Plan: Preset Bundle Service Foundation

## Objective

Make `PresetServiceMock` a deterministic real/hybrid preset service foundation: category-correct, metadata-aware, selection-persistent, and import/export validated, while preserving existing viewmodel and CLI APIs.

## Source Truth

- `third_party/OrcaSlicer/src/libslic3r/Preset.hpp`: `Preset::Type`, `is_default`, `is_system`, `is_user()`, `can_overwrite()`, `inherits`, `vendor`, `setting_id`.
- `third_party/OrcaSlicer/src/libslic3r/PresetBundle.hpp/.cpp`: system/vendor loading, selection persistence, import/export, compatibility data preservation.
- Local integration points: `src/core/services/PresetServiceMock.*`, `src/core/viewmodels/ConfigViewModel.*`, `src/qml_gui/Models/PresetListModel.cpp`, `src/cli/CliRunner.cpp`, `tests/ViewModelSmokeTests.cpp`.

## Tasks

### Task 1: Category-Correct Defaults

- Add failing tests proving `ConfigViewModel` maps printer/filament/print categories through `PresetServiceMock::PrinterCat`, `FilamentCat`, and `PrintCat` instead of raw reversed integers.
- Fix `ConfigViewModel::loadDefault`, category accessors, delete fallback, and reset-level comments/logic where category semantics are wrong.
- Preserve legacy `presetNames()` as print presets.

### Task 2: Preset Metadata API

- Add a `PresetInfo` value object exposed through service methods for name, category, built-in/read-only/user flags, inheritance parent, vendor, setting id, and value count.
- Populate metadata for builtin fallback, vendor-loaded system presets, user-created presets, renamed presets, and imported presets.
- Add tests for builtin read-only metadata and custom writable metadata.

### Task 3: Selection Persistence

- Add service-level getters/setters for selected preset by category.
- Persist selections in `QSettings` with stable keys and restore them on service construction when valid.
- Update `ConfigViewModel` to use service selections for defaults and to write selections when setters change them.
- Add tests using a temporary QSettings identity to prove restart-style persistence.

### Task 4: Validated Bundle Import/Export

- Replace placeholder all-store JSON import/export with a validated local bundle format containing version, category, metadata, and values.
- Export user presets deterministically; keep system presets read-only and not exported as user presets.
- Reject malformed imports, unknown categories, missing names, duplicate names, and missing values without mutating existing preset state.
- Add tests for a successful round trip and a malformed import no-op.

### Task 5: Phase Verification and Closeout

- Run targeted `ViewModelSmokeTests` preset tests.
- Run canonical verification: `powershell -ExecutionPolicy Bypass -File scripts/auto_verify_with_vcvars.ps1`.
- Write `44-SUMMARY.md` and `44-VERIFICATION.md` with evidence.
- Update `.planning/ROADMAP.md`, `.planning/REQUIREMENTS.md`, and `.planning/STATE.md` through GSD phase completion semantics or equivalent documented updates.

## Risks

- Vendor JSON loading is partial and not a full `PresetBundle` replacement yet; Phase 44 must preserve enough metadata for Phase 45 compatibility without over-claiming full upstream parity.
- QSettings persistence can leak between tests; tests must use unique organization/application names and clean keys.
- Import/export format is local deterministic JSON in Phase 44, not full upstream archive UI parity; Phase 48 owns the UI workflow and broader format behavior.
