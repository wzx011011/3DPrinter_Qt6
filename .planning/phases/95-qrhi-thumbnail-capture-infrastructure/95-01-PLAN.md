# Phase 95 Plan: QRhi Thumbnail Capture Infrastructure

## Objective

Replace the `RhiViewport::requestThumbnailCapture` solid-color `#18222c` PNG
stub (`RhiViewport.cpp:476-488`) with a real QRhi texture readback that reflects
the actual rendered scene (bed, plate, model mesh). Implement it via the three
Phase-94 frozen decisions:

1. **Option A offscreen RT (THUMBCAP-01):** render the scene into a separate
   offscreen `QRhiTexture` render-target at thumbnail size, then read the pixels
   back with `QRhiResourceUpdateBatch::readBackTexture()`.
2. **Single-sample offscreen RT (THUMBCAP-02):** the thumbnail RT is sample
   count 1, so no MSAA resolve step is needed at readback (the on-screen viewport
   keeps `setSampleCount(4)` per `RhiViewport.cpp:47`).
3. **Render-thread queue + queued callback (THUMBCAP-03):** mirror the existing
   `m_fitRequestCount`/`m_viewPreset` item→renderer `synchronize()` pattern
   (`RhiViewport.h:314-315`, `RhiViewport.cpp:415,435,442`,
   `RhiViewportRenderer.cpp:35`): item sets a capture request flag +
   plateIndex/size, `update()` schedules `synchronize()` to copy the request to
   the renderer, `render()` performs the offscreen render + readback, and a
   queued connection delivers the resulting QImage back to the item/GUI thread.

The offscreen RT render reuses the existing scene-render code path (bed fill +
grid lines + model mesh) parameterized by RT — not a parallel render. The
readback is asynchronous (completes on a later frame), so the renderer checks the
pending readback result at the START of the next `render()` call and emits
`thumbnailCaptured` only when the QImage is ready.

Phase 95 owns capture infrastructure only. 3MF write-side population,
save-reload round-trip, mock-path removal, and final runtime visual proof are
owned by Phases 96, 97, and 98 respectively.

## Requirements

- `THUMBCAP-01`: The Qt6 app captures a real screenshot of the current 3D
  viewport via QRhi texture readback, replacing the `requestThumbnailCapture`
  solid-color stub (`RhiViewport.cpp:476`) so the captured pixels reflect the
  actual rendered scene (bed, plate, model mesh).
- `THUMBCAP-02`: The capture handles multisampled (MSAA) render targets — the
  viewport sets sample count > 1 (`RhiViewport.cpp:47`) — so the readback
  produces a non-multisampled QImage. Frozen approach: render the thumbnail to
  a single-sample offscreen RT so NO resolve step is needed.
- `THUMBCAP-03`: The capture runs on the render thread (inside
  `RhiViewportRenderer::render`) via a cross-thread request queue
  (item → renderer), and delivers the resulting QImage back to the item/GUI
  thread via a callback, mirroring the existing `m_fitRequestCount`/
  `m_viewPreset` request pattern.
