# Phase 98 Plan: Thumbnail Verification And Cleanup

## Objective

Close the v4.3 milestone (Real Thumbnail Capture And 3MF Round-Trip) by removing
the mock thumbnail generator that has been a dead-end placeholder since Phase 95
shipped real QRhi capture, and by running the canonical end-to-end verifier.

Phases 95-97 closed the three real halves: capture (Phase 95), 3MF write
(Phase 96), and save→reload round-trip (Phase 97). The mock QPainter generator
(`ProjectServiceMock.cpp:4294 generatePlateThumbnail` / `:4414
generatePlateThumbnailVariant`) is now ONLY consumed as a fallback in the UI
plate-card (`PreparePage.qml:3157`) when no real captured thumbnail is available,
and by two legacy test slots (`thumbnailVariantsProduceValidData`,
`thumbnailRoundTrip`) that test the mock itself rather than real behavior. Phase
98 replaces the mock fallback with a new `plateThumbnailBase64` accessor that
returns the PERSISTED `PartPlate::thumbnail()` (the Phase 97 read-side artifact),
removes the mock generators, updates the legacy tests, and runs the canonical
verifier.

1. **THUMBVERIFY-01 (mock removal, no dead paths):** Replace the mock
   `generatePlateThumbnail` / `generatePlateThumbnailVariant` with a real
   `plateThumbnailBase64(plateIndex)` accessor that encodes the persisted
   `PartPlate::thumbnail()` as base64 PNG (empty when no thumbnail is cached).
   Update `EditorViewModel` (the QML-facing delegate) and `PreparePage.qml` (the
   plate-card fallback) to use it. Remove the mock generators entirely. Update
   the two legacy test slots that tested the mock to either use the new accessor
   or synthesize a QImage directly (Phase 97 pattern). No dead/disconnected
   code remains; the real QRhi capture path (Phase 95) is the sole thumbnail
   source.

2. **THUMBVERIFY-02 (canonical verifier + launch + regression-free):** Run the
   canonical verifier (`scripts/auto_verify_with_vcvars.ps1`), confirm
   `OWzxSlicer.exe` builds and the process launches, confirm Prepare/Preview/
   AssembleView rendering is regression-free (regression ctest:
   PartPlateTests, ViewModelSmokeTests, QmlUiAuditTests, PrepareSceneDataTests),
   and confirm runtime thumbnail capture is reachable (the `requestGLThumbnail`
   → `requestThumbnailCapture` → `deliverThumbnail` → `lastThumbnailData` chain
   is intact after mock removal). Per STATE.md deferred items, runtime VISUAL
   evidence (Windows capture API) is blocked; reachability is verified via
   process-liveness + canonical verifier + regression ctest, not screenshots.

Phase 98 also addresses the Phase 97 REVIEW MEDIUM-3 multi-plate write-side gap
(documented, with the minimal fix applied if scope allows) and removes the
scratch build logs + scripts that accumulated across Phases 95-97 (verified
unreferenced before deletion).

## Security Threat Model (ASVS L1)

This phase removes dead mock code and swaps a UI fallback source. Threat surface
assessment:

- **No new attack surface introduced.** The changes are: (1) remove a QPainter
  placeholder generator, (2) add a read-only accessor that encodes an in-memory
  QImage to base64 PNG, (3) swap a QML method name. No network, no external
  input parsing, no authentication, no file-system writes, no privilege
  boundaries crossed.
- **`plateThumbnailBase64` reads only trusted internal state** —
  `PartPlate::thumbnail()` is populated exclusively by the Phase 95 capture
  path (in-process QRhi readback) and the Phase 97 read-side helper (decodes PNG
  bytes from the project's own 3MF archive). No untrusted external input reaches
  it.
- **No injection vectors.** The base64 PNG encoding uses `QImage::save` to a
  `QBuffer` (in-memory); the output is consumed by QML `Image.source` which Qt
  decodes safely. No shell interpolation, no SQL, no eval.
- **Multi-plate write-side fix (TV-10, if applied):** adds more `ThumbnailData`
  entries to the existing `StoreParams::thumbnail_data` vector; this is the same
  data structure Phase 96 already writes, just with more entries. No new file
  format, no new parser.

Threats open: 0. No blocking (high-severity) threats. The phase is a cleanup +
verification pass; security posture is unchanged or improved (removing dead code
reduces surface).
