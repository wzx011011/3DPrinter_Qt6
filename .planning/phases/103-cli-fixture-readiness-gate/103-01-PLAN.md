# Phase 103 Plan: CLI Fixture Readiness Gate

## Objective

Close FIXTURE-02 by replacing the current `QTimer::singleShot(0)` startup-fixture gate at `main_qml.cpp:213` with a deterministic two-stage wait on `QQmlApplicationEngine::objectCreated` (QML object tree built) AND `QQuickWindow::frameSwapped` (first frame rendered). The current `singleShot(0)` runs on the next event-loop iteration but before the scene graph has necessarily rendered a frame, so screenshots captured by an external tool may catch a blank/partial window. The two-stage gate guarantees both the QML object tree exists AND a frame has been painted — closing the long-standing Windows-capture-API runtime-evidence blocker (the canonical workaround bar per STATE.md deferred items).

**Key finding from pre-planning research:** the argv plumbing ALREADY EXISTS end-to-end — `parseStartupOpenRequest` (main_qml.cpp:170-205) reads the 4 QCommandLineOption values, `applyStartupOpenRequests` (main_qml.cpp:207-257) applies them, and the call site at `main_qml.cpp:349` invokes it after `engine->load()`. Phase 103 changes ONLY the gate mechanism inside `applyStartupOpenRequests` — no new flags, no new plumbing. This is the smallest possible fix that closes FIXTURE-02.

## Requirements

- `FIXTURE-02`: The existing argv plumbing (`--load-model`, `--open-page`, `--open-dialog`, `--skip-first-run` at `main_qml.cpp:170-257`) is gated on `QQmlApplicationEngine::objectCreated` + `QQuickWindow::frameSwapped` (not the current `singleShot(0)` trick) so screenshots are deterministic — closing the long-standing Windows-capture-API runtime-evidence blocker.

          // One-shot connection: disconnect in the handler so it fires exactly once.
