# Phase 92 Plan: Assembly Measurement Gizmo

## Objective

Port the Assembly measurement gizmo (`Ctrl+Y`, `GLGizmoAssembly` /
`GLGizmoMeasure` `ONLY_ASSEMBLY` mode) to the Qt6/RHI `CanvasAssembleView` host,
delivering the three screenshot-visible behaviors from `shotScreen/装配页_测量.png`:

1. A gizmo invokable via `Ctrl+Y` on AssembleView, with activability mirroring
   upstream `GLGizmoAssembly::on_is_activable()`
   (`GLGizmoAssembly.cpp:53-68`: explosion ratio ≈ 1.0 + ≥2 volumes selected).
2. A measurement between the two selected volumes — distance + angle — computed
   by a C++ helper (NOT in QML, per project architecture), surfaced to the
   renderer and the panel.
3. Measurement overlay rendering (white dashed dimension line with arrowheads +
   teal value annotation box) in the `CanvasAssembleView` branch of
   `RhiViewportRenderer`, plus a right-side "测量" panel in `AssemblePage.qml`.

Out of scope for Phase 92 (deferred or simplified — see Scope Decision below):
full per-triangle feature picking (point/edge/circle/plane), face-to-face
assembly transforms, distance-XYZ editing, the data pool (Phase 93), and runtime
pixel proof (Phase 93).

## Requirements

- `ASMMEASURE-01`: User can invoke the Assembly measurement gizmo (`Ctrl+Y`) on
  the AssembleView canvas, mirroring `GLGizmoAssembly` activability rules
  (explosion ratio near 1.0, multi-volume selection). Covers matrix region
  `ASM-ASSEMBLY-GIZMO`.
- `ASMMEASURE-02`: The Assembly gizmo measures distances/relations between
  selected volumes using the upstream `ONLY_ASSEMBLY` measure mode, rendering
  measurement overlays anchored to the right geometry. Covers region
  `ASM-MEASURE-OVERLAY`.

## Scope Decision (what is ported vs. simplified) — EXPLICIT

Upstream `GLGizmoMeasure` (`third_party/OrcaSlicer/src/slic3r/GUI/Gizmos/
GLGizmoMeasure.cpp`, ~2670 lines) is a large feature-picking engine: it
raycasts into the mesh, extracts `Measure::SurfaceFeature`s (point/edge/circle/
plane) via `Measure::Measuring` (which requires a per-volume
`indexed_triangle_set`), computes `Measure::MeasurementResult` (distance
strict/infinite/xyz + angle between edges/planes), renders dimensioning in
screen-space with ImGui value boxes, and exposes face-to-face / point-to-point
assembly transforms (parallel, center-coincidence, reverse-rotation,
around-center) that mutate the model.

**Porting that engine wholesale in one phase is out of proportion and has a
hard dependency that is not yet plumbed**: it needs (a) per-volume
`indexed_triangle_set` access through `ProjectServiceMock` (the mesh blob is
GPU-oriented TLV, not an ITS), (b) a scene raycaster on the RHI path
(none exists — Prepare picks by source-object, not by triangle), and (c) the
`AssembleViewDataPool` (explicitly Phase 93 / ASMROUTE-02). The screenshot
evidence does not require the full engine: `装配页_测量.png` shows one dimension
line + one value box + the panel — the common-case distance/angle measurement
between two selected volumes.

Phase 92 therefore delivers a **minimal-but-correct measurement that matches the
screenshot and the activability rules**, with the internal engine simplified and
documented as a deviation:

| Capability | Upstream | Phase 92 | Rationale |
|---|---|---|---|
| Activability gate (`Ctrl+Y`) | `GLGizmoAssembly.cpp:53-68` | **FULL** | Source-truth rule; cheap; ASMMEASURE-01 success criterion. |
| Measure mode = `ONLY_ASSEMBLY` | `GLGizmoAssembly.cpp:25-29` | **FULL** (enum value + gate) | Distinguishes Assembly from Prepare measure; ASMMEASURE-01. |
| Distance between 2 selected volumes | center-to-center + XYZ | **SIMPLIFIED**: AABB-center to AABB-center + per-axis XYZ + straight (Euclidean) distance | Correct for the common multi-volume case; screenshot shows one distance value. Full feature-picking (point/edge/face) needs ITS + raycaster (future). |
| Angle between 2 selected volumes | edge/plane dihedral angle | **SIMPLIFIED**: angle between the two volumes' dominant AABB axes (longest axis direction) | Produces the `90.000°`-style value box; no plane extraction. Documented as approximation. |
| Overlay: white dashed dimension line + arrowheads | screen-space `dashed_lines` shader | **MATCHING**: dashed GL_LINES segments (reuse Phase 91 connector pattern) + 2 arrowhead triangles at endpoints, white | Matches `装配页_测量.png` visuals; no new shader. |
| Overlay: teal value annotation box | `ImDrawList::AddRectFilled` semi-transparent white + ImGui text | **MATCHING**: value rendered in the 测量 panel (QML) AND a teal value box drawn in-world at the dimension midpoint via a translucent fill pipeline | Screenshot shows teal value box at the line midpoint. |
| Right-side 测量 panel | `on_render_input_window` ImGui | **MATCHING**: QML panel on the right of AssemblePage | Screenshot position; project architecture (QML for presentation). |
| Plane-selection indicators ("选中 N 平面") | ImGui | **SIMPLIFIED**: static "选中 1 平面"/"选中 2 平面" label reflecting the two selected volumes | Screenshot-visible; full plane picking is future. |
| Assembly transforms (parallel / coincidence / reverse / around-center) | `set_to_parallel` etc. | **DEFERRED** (future) | Not screenshot-visible; needs full feature engine. |
| Feature reset / re-pick / mode combo (Face-Face / Point-Point) | ImGui | **DEFERRED** (future) | Not required by screenshot; needs feature engine. |
| Distance-XYZ editing (move volumes) | `set_distance` | **DEFERRED** (future) | Mutates model; out of scope for measurement display. |

**This is the documented simplification.** It honors the phase goal (screenshot
parity + activability + correct distance/angle for the common case) while
keeping the work proportional. Everything deferred is recorded as a future
enhancement that requires the ITS + raycaster + data-pool prerequisites.

## Locked Decisions (from 92-CONTEXT.md + codebase — MUST honor)

1. **Gizmo mode enum value**: add `GizmoAssemblyMeasure` to `RhiViewport::GizmoMode` (after `GizmoSlaSupports = 18`, value `19`). The existing `GizmoMeasure = 3` is the Prepare measure gizmo (`Ctrl+U` in PreparePage.qml:158); Assembly measure is a distinct upstream gizmo (`Ctrl+Y`, `GLGizmoAssembly`) and MUST get its own enum value so the two never collide.
2. **Activability**: `EditorViewModel::availableGizmoMask()` AssembleView early-return (EditorViewModel.cpp:2026 `if (m_activeCanvasType == 2) return 0;`) is replaced with a gate that returns the Assembly-measure bit ONLY when `explosionRatio ≈ 1.0 && selectedObjectCount >= 2` (mirrors `GLGizmoAssembly.cpp:53-68`). On Prepare/Preview the mask is unchanged (the gate is AssembleView-only).
3. **Measurement geometry stays in C++**: a new helper `AssemblyMeasureGeometry` in `src/core/rendering/` computes distance/angle from two volume AABBs. NOT in QML (AGENTS.md: business logic in core/). Pure data struct, unit-testable without a renderer.
4. **Overlay renders in the renderer**: the `CanvasAssembleView` branch of `RhiViewportRenderer` draws the dashed dimension line + arrowheads + teal value box, gated to `m_gizmoMode == GizmoAssemblyMeasure && m_canvasType == CanvasAssembleView`. Reuses Phase 91's connector-line buffer pattern + the existing translucent fill pipeline.
5. **Panel in QML**: the right-side "测量" panel lives in `AssemblePage.qml`, bound to viewmodel properties. No business logic in inline scripts beyond routing.
6. **Ctrl+Y shortcut**: `AssemblePage.qml` handles `Ctrl+Y` → `editorVm.activateAssemblyMeasureGizmo()` (Q_INVOKABLE that sets the gizmo mode if activable). Mirrors PreparePage.qml:156-159 `Ctrl+U`→`GizmoMeasure` pattern.
7. **Two selected volumes source**: the existing `selectedObjectCount` (count of selected source objects) is the activability signal. The measurement uses the first two selected source objects' AABB centers from `m_prepareScene.modelBatches()` (each batch carries `bounds` + `sourceObjectIndex`).

## Per-Volume Measurement Geometry (key implementation finding)

The measurement needs two AABBs. After Phase 91's per-volume blob restructure
(`ProjectServiceMock::meshData` emits one `ObjBatch` per volume), each
`PrepareSceneData::ModelBatch` already carries per-volume `bounds` +
`sourceObjectIndex`. So the measurement can be computed entirely from data the
renderer already has — NO mesh-ITS access, NO raycaster, NO new service surface.
The two selected source objects' first volume batches supply the two AABBs;
their centers give the distance, and their longest-axis directions give the
angle. This is the load-bearing simplification that keeps Phase 92 proportional.
