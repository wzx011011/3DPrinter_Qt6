# Plan 24-02: Prepare Bed And Plate Context Binding

<objective>
Expose the C++ bed and active plate context needed by the QRhi Prepare renderer and bind it into the viewport without moving filtering or scene conversion into QML. This plan connects the Plan 01 scene contract to `EditorViewModel`, `RhiViewport`, and `PreparePage.qml`.
</objective>

  <task id="24-02-T2" type="auto" tdd="true">
    <name>Expose renderer-facing active plate context from EditorViewModel</name>
    <files>
src/core/viewmodels/EditorViewModel.h
src/core/viewmodels/EditorViewModel.cpp
tests/ViewModelSmokeTests.cpp
    </files>
    <behavior>
`activePlateObjectIndices` changes when `setCurrentPlateIndex()` changes the plate, when loaded project/plate membership changes, or when all project state is cleared. It must be derived from `ProjectServiceMock::currentPlateObjectIndices()` or equivalent `PartPlateList` truth.
    </behavior>
    <action>
Add a renderer-facing `Q_PROPERTY(QVariantList activePlateObjectIndices READ activePlateObjectIndices NOTIFY stateChanged)` or equivalent invokable on `EditorViewModel`. Keep the conversion simple and typed: C++ receives `QList<int>` or `QVector<int>` from the service and returns a `QVariantList` for QML binding. Do not parse `meshData()` and do not filter by QML. Keep existing object-list behavior intact unless tests prove it must change.
    </action>
    <verify>
      <automated>build/ViewModelSmokeTests.exe activePlateObjectIndicesFollowCurrentPlateWithoutFallback</automated>
    </verify>
    <done>
`EditorViewModel` provides a stable C++ plate membership surface for the renderer and emits `stateChanged()` through existing mutation paths.
    </done>
  </task>

Add bed and plate properties to `RhiViewport`: `bedWidth`, `bedDepth`, `bedOriginX`, `bedOriginY`, `bedShapeType`, `bedDiameter`, `currentPlateIndex`, `plateCount`, and `activePlateObjectIndices`. Pass synchronized values to `RhiViewportRenderer` and then to `PrepareSceneData`. Update `PreparePage.qml` so `GLViewport` binds directly to `root.editorVm` properties. Extend `QmlUiAuditTests` to assert the bindings exist and that `PreparePage.qml` does not filter, loop over, or parse renderer object membership in QML.
    </action>
    <verify>
      <automated>build/QmlUiAuditTests.exe prepareViewportBindsBedAndPlateContext</automated>
    </verify>

<threat_model>
| Threat | STRIDE | Mitigation | Disposition |
|--------|--------|------------|-------------|
| QML computes renderer membership and diverges from C++ plate truth | Tampering | Static audit requires direct binding and forbids QML filtering/parsing | mitigate |
| Active plate switch leaves stale renderer data | Denial of Service | Setters call `update()` and synchronization passes all plate context into `PrepareSceneData` | mitigate |
| Display fallback leaks inactive object indices into active renderer scene | Information Disclosure | Separate renderer-facing active list from UI display fallback and test the empty case | mitigate |
</threat_model>
