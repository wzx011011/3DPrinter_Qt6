## Non-blocking Gaps

### D3D12 Prepare render crash (the only genuine gap)
D3D12 backend crashes (Segfault) during Prepare rendering on startup. Root cause is in Phase 23-25's Prepare buffer/pipeline code, NOT in Phase 26's Preview pipeline (verified by reverting Phase 26 and still crashing). BUG-V31-1 (beginPass ordering) was fixed but didn't resolve it. **D3D11-first workaround** in RhiBackendSelector protects all users (OWZX_RHI_RENDERER=1 → D3D11 safe default). Deep D3D12 investigation deferred to v3.2.

### Preview not visually verified under QRhi
Phase 26 code is correct (GCV1 parsing, draw-range, color modes, travel toggle) and render_bench validates the raw pipeline at 1M/5M segments. But the actual app Preview page hasn't been visually verified with OWZX_RHI_RENDERER=1 (requires: slice a model → switch to Preview → confirm segments render). This is a manual verification item, not a code gap.

## Top 5 Risks for v3.2

1. **D3D12 crash** — root cause unknown; users on D3D12-only hardware (rare on Windows) can't use QRhi. D3D11-first is a safe default.
2. **Preview not visually verified** — the QRhi Preview path is code-complete but never been seen rendering in the actual app.
3. **Marker geometry missing on QRhi** — tool-position marker not ported to QRhi (D-26-04 deferred).
4. **Upload timing = 0** — m_previewLastUploadMs is always 0 (deferred upload); real timing only in render_bench.
5. **moveEnd proportional clamp** — playback position approximates, not per-move precise.

## Conclusion

v3.1 established the QRhi high-performance rendering foundation for Prepare and Preview. D3D11 is the safe working backend with interactive 1M/5M segment performance. D3D12 crashes (workaround in place). Code review caught and fixed 2 issues (BUG-V31-1 + DESIGN-V31-4). The milestone's definition of done is met at code/benchmark level; visual verification + D3D12 fix are v3.2 items.
